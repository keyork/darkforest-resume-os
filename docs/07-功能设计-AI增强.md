# 07 · 功能设计：AI 增强功能

本文档覆盖四个基于 AI 的功能增强方向：
1. 档案条目润色
2. 简历健康度评分
3. 批量 JD 对比分析
4. 面试问题预测

所有功能复用现有的 `callAgent` / `callAgentText` 调用链，通过前端传入的 AI 设置头发起请求。

---

## 一、档案条目 AI 润色

### 背景

用户手动录入或从旧简历导入的条目描述，往往缺少量化数据、动词不够有力、STAR 结构不完整。目前改写只能靠用户自己重新编辑，没有 AI 辅助。

### 功能设计

**入口：** ItemCard 的 hover 菜单里新增 `✨ 润色` 按钮（在编辑、删除旁边）

**触发流程：**
1. 用户点击润色按钮
2. 在 ItemCard 下方展开一个"润色工作区"面板（不打开新弹窗，减少上下文切换）
3. 左侧显示原始内容，右侧显示 AI 重写版本（加载中时显示骨架屏）
4. 用户可点击"采用"（替换原内容）或"忽略"（关闭面板）
5. 可选：支持"再试一次"重新生成

**支持的条目类型：**
- `experience`：润色每条 achievement，也可润色整体 description
- `project`：润色项目描述和亮点
- `skill`：不适用（技能条目结构化程度高，不需要润色）
- `education` / `certification`：不适用

**Prompt 策略（experience 示例）：**

```
System:
你是一位专业的简历顾问，擅长用 STAR（情境-任务-行动-结果）原则和量化表达优化简历描述。

User:
请润色以下工作经历描述，要求：
1. 动词开头，使用有力的行动动词
2. 尽量量化结果（如"提升 30%"、"覆盖 10 万用户"）
3. 保持原意，不要编造数据，如无法量化则保留原表述
4. 中文输出，控制在 50-80 字以内

原文：{originalText}
```

**API 端点：** `POST /api/items/[id]/polish`

**返回格式：**
```json
{ "polished": "string" }
```

## 二、简历健康度评分

### 背景

用户刚建好档案，还没有目标 JD 时，很难判断"我的档案质量怎么样"。现有功能需要先有 JD 才能分析，这是一个空窗期。

简历健康度评分在**无 JD** 的前提下，直接对档案做质量诊断。

### 评估维度

| 维度 | 说明 | 满分 |
|---|---|---|
| 完整度 | 五类条目（技能/经历/项目/教育/证书）是否都有填写 | 20 分 |
| 量化程度 | 经历和项目描述中包含数据/指标的比例 | 25 分 |
| 技术关键词密度 | 技能标签和描述中的技术词汇丰富度 | 20 分 |
| 时间线连贯性 | 经历是否存在明显空白期，时间是否从近到远排列 | 15 分 |
| 表达清晰度 | 描述是否有动词、是否避免了模糊词（"负责了""参与了"等）| 20 分 |

总分 100 分，<60 提示需要较大改进，60-79 提示有明显短板，≥80 提示档案质量良好。

### 页面入口

两处入口：
1. **Profile 页顶部** — 展示一个小分数卡片，点击展开详情
2. **Dashboard** — 在状态信息区展示"档案得分：XX 分"，旁边有"查看诊断"链接

### 详情展示

```
┌────────────────────────────────────┐
│  档案健康度   78 分  ██████████░░  │
│                                    │
│  完整度        18/20   ✓           │
│  量化程度      15/25   ⚠ 建议改进  │
│  技术关键词    18/20   ✓           │
│  时间线连贯性  12/15   ✓           │
│  表达清晰度    15/20   ⚠ 建议改进  │
│                                    │
│  改进建议：                         │
│  · 3 条工作经历描述缺少量化数据      │
│  · 发现"负责""参与"等模糊动词 5 处  │
└────────────────────────────────────┘
```

**API 端点：** `POST /api/profile/health-check`

**返回格式：**
```json
{
  "total": 78,
  "breakdown": {
    "completeness": { "score": 18, "max": 20, "note": "..." },
    "quantification": { "score": 15, "max": 25, "note": "..." },
    "keywords": { "score": 18, "max": 20, "note": "..." },
    "timeline": { "score": 12, "max": 15, "note": "..." },
    "clarity": { "score": 15, "max": 20, "note": "..." }
  },
  "suggestions": ["string"]
}
```

> **注意：** 健康度评分可以在不调用 AI 的情况下做部分计算（完整度、时间线），只有量化程度、表达清晰度需要 LLM 判断。可以先做规则计算，再用 AI 补充评估。

## 三、批量 JD 对比分析

### 背景

用户通常同时在考虑多个岗位，需要决策"先投哪个"。目前只能逐个分析，效率低，且无法横向比较。

### 功能设计

**入口：** JD 匹配页新增"批量对比"模式切换

**操作流程：**
1. 用户在已解析的 JD 列表中多选（最多 5 个）
2. 点击"批量分析"，并行发起多个匹配请求
3. 分析完成后展示对比表格

**对比维度：**

```
               字节-后端   腾讯-前端   阿里-算法   网易-数据
────────────────────────────────────────────────────────
综合匹配度        82          71          68          79
技术匹配          85          72          55          81
经验匹配          80          75          70          78
差距数量（高优）    1           2           4           1
需要补足技能       无         React        LeetCode    Spark
推荐优先级        ★★★         ★★          ★           ★★★
```

**推荐优先级逻辑：**
综合考虑：匹配度得分 + 高优差距数量 + 是否有匹配历史

**API 端点：** `POST /api/match/batch`

**请求：**
```json
{ "jdIds": ["id1", "id2", "id3"] }
```

**返回：**
```json
{
  "results": [MatchResult],
  "comparison": {
    "ranking": ["id1", "id3", "id2"],
    "summary": "string"
  }
}
```

**实现注意：**
- 并行发起多个匹配请求（`Promise.allSettled`），单个失败不影响整体
- 批量匹配消耗较多 token，需要在 UI 上提示预估时间
- 结果自动保存到历史匹配记录中（和单次分析共用数据表）

## 四、面试问题预测

### 背景

用户完成 JD 匹配分析后，下一步是准备面试。但"差距分析"告诉的是"你哪里不足"，不是"面试官会问什么"。这两者之间需要一层转化。

### 功能设计

**入口：** 匹配结果页新增"面试准备"Tab（与"需求匹配""差距分析"并列）

**生成内容结构：**

```
面试准备建议（基于此次匹配）

【必考题】（来自高优差距项）
Q: 你在 Kubernetes 方面的经验怎么样？
→ 你的背景：有 Docker 使用经验，但无 K8s 实际部署经验
→ 回答思路：坦诚学习状态 + 展示快速学习能力 + 相关项目迁移经验

Q: 描述一次你优化系统性能的经历
→ 你的背景：XX 项目有 30% 接口优化记录
→ 回答思路：使用 STAR 结构，重点量化优化结果

【常规技术题】（来自强匹配技术要求）
Q: 讲讲你对 React 状态管理的理解？
→ 你的背景：3 年 React 经验，使用过 Redux 和 Zustand
→ 回答思路：对比各方案优劣，结合实际选型经验
...

【文化/行为题】
Q: 描述一次你与团队产生分歧并最终解决的经历
→ 建议结合 [XX 项目] 中的协作经历展开
```

**API 端点：** `POST /api/match/[id]/interview-prep`

**返回格式：**
```json
{
  "mustAsk": [
    {
      "question": "string",
      "background": "string",  // 用户的背景信息
      "answerHint": "string"   // 回答思路
    }
  ],
  "technical": [...],
  "behavioral": [...]
}
```

**Prompt 关键输入：**
- 用户档案摘要（技能列表 + 经历摘要）
- 匹配结果（差距项 + 强匹配项 + JD 文化关键词）

**缓存策略：** 同一个 matchResultId 的面试准备结果缓存到数据库，避免重复生成。

---

## 共同技术注意点

1. **Token 消耗控制**：润色和健康度评分输入较小；批量 JD 和面试准备输入较大，应在 UI 上有耗时提示
2. **流式输出**：面试准备内容较长，建议使用 `stream: true` + SSE 流式输出，提升体验
3. **错误兜底**：所有 AI 功能需要处理 "AI 未配置" 的情况，引导用户去设置页
4. **重试机制**：润色和面试准备都是用户明确主动触发的，失败后直接展示错误信息即可，无需自动重试
