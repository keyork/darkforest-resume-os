# 08 · 功能设计：数据管理

本文档覆盖四个数据层面的功能：
1. 数据导出与备份
2. PDF 直接导出
3. 全局搜索
4. 对话式档案录入

---

## 一、数据导出与备份

### 背景与风险

当前所有档案数据存储在 SQLite 文件 `./db/resume-agent.db` 中。对普通用户（尤其是本地部署用户）而言，这个文件几乎不可见，迁移机器、重新安装或误删都会导致数据丢失。这是当前架构最大的可用性风险。

### 导出功能

**入口：** 设置页新增"数据管理"Card，包含"导出全部数据"按钮

**导出格式：JSON**

```json
{
  "exportedAt": "2025-03-03T12:00:00.000Z",
  "version": "1.0",
  "profile": { ... },
  "items": [...],
  "jobDescriptions": [...],
  "matchResults": [...],
  "generatedResumes": [...]
}
```

> `generatedResumes` 导出时包含完整 `content`（Markdown 全文），因此文件可能较大（数十 KB 到几百 KB）。

**实现：**
- 新建 `GET /api/export` 路由，从数据库读取全量数据，返回 JSON
- 前端收到后用 `URL.createObjectURL` 触发浏览器下载，文件名格式：`darkforest-backup-{YYYY-MM-DD}.json`

### 导入/还原功能

**入口：** 同一个"数据管理"Card，"从备份还原"按钮

**操作流程：**
1. 用户选择一个 `.json` 备份文件
2. 前端展示文件摘要（导出时间、各类数据数量）
3. 用户确认"覆盖现有数据"或"合并"（合并：以 ID 为准，存在则跳过，不存在则插入）
4. `POST /api/import` 上传 JSON，服务端处理导入逻辑

**注意事项：**
- 导入时应做 JSON Schema 校验，版本不匹配时提示风险
- "覆盖"模式需要二次确认弹窗
- 导入完成后触发 React Query 全量失效刷新

## 二、PDF 直接导出

### 背景

当前"生成简历"页只输出 Markdown，用户需要自行转 PDF（通常需要安装插件或复制到第三方工具）。这是完成"最后一公里"的关键功能。

### 方案选型

| 方案 | 优点 | 缺点 | 推荐度 |
|---|---|---|---|
| 浏览器打印 CSS | 零依赖，100% 免费 | 样式控制有限，用户体验不够好 | ★★★ |
| `@react-pdf/renderer` | 样式控制精确，纯前端 | bundle 体积大（~300KB），学习成本高 | ★★ |
| `html2canvas` + `jsPDF` | 所见即所得 | 图片化 PDF，不可搜索，质量差 | ★ |
| 服务端 Puppeteer | 输出质量最好 | 需要 Chromium 环境，部署复杂 | ★ |

**推荐方案：浏览器打印 CSS（短期） + `@react-pdf/renderer`（中期）**

### 浏览器打印方案（快速实现）

1. 在 `ResumePreview` 组件里渲染 Markdown 为 HTML（现有功能）
2. 新建一个隐藏的 `<iframe>` 或弹出窗口，注入简历 HTML + 打印样式
3. 调用 `window.print()`，系统打印对话框里用户选"另存为 PDF"

打印样式要点：
```css
@media print {
  body { font-family: 'Times New Roman', serif; font-size: 11pt; }
  h1   { font-size: 16pt; }
  h2   { font-size: 13pt; border-bottom: 1px solid #000; }
  @page { margin: 1.5cm; size: A4; }
}
```

**ExportButtons 组件改造：** 在现有"复制 Markdown"旁边加"下载 PDF"按钮。

### `@react-pdf/renderer` 方案（完整实现）

设计一套简历 PDF 模板组件（`<Document>`, `<Page>`, `<Text>` 等），将 Markdown 解析为 PDF 布局。

优先实现一套模板（简约单栏），后续可以扩展为多套风格（双栏、带侧边栏等）。

## 三、全局搜索

### 背景

随着档案数据增多（技能几十条、项目多个、JD 积累），用户需要快速定位特定内容（如"我在哪个项目里用过 Kafka"）。

### 功能设计

**触发方式：** `⌘K`（Mac）/ `Ctrl+K`（Windows），或侧边栏顶部搜索图标

**搜索范围：**

| 类型 | 搜索字段 | 跳转目标 |
|---|---|---|
| 技能 | 名称、分类 | `/profile?tab=skill` |
| 工作经历 | 公司、职位、描述、成果 | `/profile?tab=experience` |
| 项目 | 名称、描述、技术栈 | `/profile?tab=project` |
| 教育 | 学校、专业 | `/profile?tab=education` |
| 证书 | 名称、发证机构 | `/profile?tab=certification` |
| JD | 公司、岗位、原始文本 | `/match`（选中对应 JD） |
| 生成简历 | 策略标签、内容片段 | `/generate`（选中对应简历） |

**技术选型：**

使用 `fuse.js` 做客户端模糊搜索，全量数据在应用启动时（或首次搜索时）从 API 加载一次，存在 React Query 缓存中。

```ts
import Fuse from 'fuse.js';

const fuse = new Fuse(allItems, {
  keys: ['name', 'description', 'company', 'position'],
  threshold: 0.3,
  includeMatches: true,  // 高亮匹配片段
});
```

**UI 设计：** 居中弹出的命令面板（Command Palette 风格），分组展示搜索结果：

```
┌──────────────────────────────────────────┐
│ 🔍  搜索档案、JD、简历…           ⌘K    │
├──────────────────────────────────────────┤
│  技能 (2)                                │
│  ▸ React     前端框架  ████             │
│  ▸ Kafka     消息队列  ████             │
│                                          │
│  工作经历 (1)                            │
│  ▸ XX 公司 · 后端工程师 · 2022-2024     │
│    "负责 Kafka 消息队列的架构设计…"      │
└──────────────────────────────────────────┘
```

**实现组件：** 可以使用 `cmdk` 库（shadcn 的 Command 组件底层就是它），配合 `Dialog`。

## 四、对话式档案录入

### 背景

手动填写档案表单是当前录入体验的主要摩擦点。用户往往知道自己"做了什么"，但不清楚要填哪个字段、怎么描述最好。

对话式录入允许用户用自然语言描述经历，AI 自动解析成结构化条目。

### 功能设计

**入口：** Profile 页顶部或每个 Tab 里新增"快速录入"按钮，打开侧边 Sheet

**交互流程：**

```
用户：我在字节跳动做了两年后端，主要负责推荐系统的服务端开发，
     带着 3 个人，2023 年 7 月到 2025 年 6 月

AI 解析预览：

  ✦ 工作经历
  ┌────────────────────────────┐
  │ 公司：字节跳动              │
  │ 职位：后端工程师（待确认）   │
  │ 时间：2023-07 ~ 2025-06    │
  │ 描述：推荐系统服务端开发     │
  │ 团队规模：3人               │
  └────────────────────────────┘

  ✦ 技能（自动推断）
  ┌────────────────────────────┐
  │ 推荐系统  · 后端  · 待确认  │
  └────────────────────────────┘

[编辑并确认]  [直接添加]  [重新描述]
```

**多轮补充：**

```
AI：有没有具体的技术成果或量化数据？比如性能提升、用户覆盖规模等

用户：推荐点击率提升了 18%，服务 QPS 从 5000 提升到 12000

AI：已更新经历描述，是否还需要补充？
```

**API 端点：** `POST /api/profile/quick-add`（两步式：先解析预览，再确认写入）

**Prompt 关键设计：**

第一步（解析）：给定用户自然语言，返回结构化的条目 JSON，字段尽量填充，无法推断的打上 `"_uncertain": true` 标记

第二步（补充）：如果用户继续对话，在上一步结果基础上追加/修正字段

**与现有导入功能的区别：**

| | 现有简历导入 | 对话式录入 |
|---|---|---|
| 输入 | PDF/DOCX 文件 | 自然语言 |
| 粒度 | 整份简历一次性解析 | 逐条、逐段录入 |
| 控制感 | 批量确认 | 即时预览 + 逐条确认 |
| 适用场景 | 有旧简历文件 | 没有文件 / 补充新经历 |

## 依赖与共享组件

| 功能 | 新增依赖 |
|---|---|
| 数据导出 | 无（原生 `Blob` + `URL.createObjectURL`） |
| PDF 导出（打印方案） | 无 |
| PDF 导出（renderer 方案） | `@react-pdf/renderer` |
| 全局搜索 | `fuse.js`（~25KB，已有 `cmdk` via shadcn） |
| 对话式录入 | 无新依赖，复用现有 AI 调用链 |
