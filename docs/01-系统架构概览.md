# 01-系统架构概览

## 1. 项目目标

`darkforest-resume-os` 的核心目标不是“生成一份静态简历”，而是将候选人的职业经历拆成结构化能力单元，然后在目标岗位上下文中进行：

- 结构化存储
- 语义匹配
- 策略分析
- 多版本简历生成

这使得系统更接近一个“履历操作系统”，而不是一次性的文档工具。

## 2. 技术栈

当前实现使用以下技术：

| 层 | 技术 |
| --- | --- |
| Web 框架 | Next.js 14 App Router |
| 语言 | TypeScript |
| 状态管理 | TanStack Query v5 |
| 参数同步 | nuqs |
| UI | Tailwind CSS + shadcn/ui + lucide-react |
| 数据库 | SQLite |
| ORM | Drizzle ORM |
| AI SDK | OpenAI SDK |
| 文件解析 | `pdf-parse`、`mammoth` |
| 图表 | Recharts |

## 3. 总体分层

项目可以分为 5 个层次：

### 3.1 页面层

位于 `src/app`，负责页面路由与页面级布局。

主要页面：

- `/`：Dashboard
- `/profile`：档案编辑
- `/match`：JD 匹配分析
- `/generate`：简历生成
- `/settings`：AI 设置页

### 3.2 组件层

位于 `src/components`，负责页面组合、交互控件和任务面板。

主要分组：

- `profile/`：档案管理相关组件
- `match/`：JD 匹配分析组件
- `generate/`：简历生成组件
- `layout/`：导航栏、主题切换
- `agent/`：全局任务状态管理与右侧任务面板
- `ui/`：shadcn/ui 封装组件

### 3.3 Hook 层

位于 `src/lib/hooks`，封装与 API 的请求和缓存失效逻辑。

包括：

- `useProfile`
- `useItems`
- `useJD`
- `useMatch`
- `useGenerate`

这一层负责：

- 页面与后端 API 解耦
- Query Key 统一管理
- Mutation 后自动失效缓存
- 在 AI 相关操作中接入全局任务系统

### 3.4 AI 层

位于 `src/lib/ai`。

主要职责：

- 定义 system prompt
- 统一封装 LLM 调用
- 提供四个 Agent：
  - 简历解析 Agent
  - JD 解析 Agent
  - 匹配分析 Agent
  - 简历生成 Agent

### 3.5 数据层

位于 `src/lib/db`。

主要职责：

- 初始化 SQLite 连接
- 暴露 Drizzle 实例
- 定义 schema
- 执行 DB 记录与业务对象之间的转换

## 4. 关键设计原则

## 4.1 单默认档案模型

当前实现使用固定档案 ID：

`profile_default`

这意味着当前版本是单用户、单档案模型。所有 Item、匹配结果、生成结果都围绕这个默认档案组织。

优点：

- 逻辑简单
- API 不需要用户态上下文
- 前端实现成本低

限制：

- 不支持多份候选人档案并存
- 不支持多用户隔离

## 4.2 一切履历皆结构化 Item

所有履历内容都被拆成统一 Item：

- skill
- experience
- project
- education
- certification

统一能力包括：

- 可见 / 隐藏
- 同类排序
- 手动创建或 AI 来源标记
- JSON 数据存储

## 4.3 AI 只负责“推理”，数据库负责“持久化”

AI 层不直接写数据库。

典型流程是：

1. API Route 取到输入
2. Route 调用 AI Agent
3. Agent 返回结构化结果
4. Route 负责结果入库
5. 前端刷新查询缓存

这种分离保证：

- LLM 输出可控
- 业务写库逻辑集中在 API route
- 更容易替换模型供应商

## 4.4 前端驱动 AI 配置

当前版本不再从 `.env` 读取 AI 的 `API_KEY` 和 `BASE_URL`。

实际机制是：

1. 用户在 `/settings` 页面输入配置
2. 配置存入浏览器 `localStorage`
3. 前端请求时将配置写入请求头
4. 后端从请求头中读取 AI 客户端配置

这意味着：

- AI 配置与部署环境解耦
- 不同浏览器实例可以连接不同模型服务
- 服务端本身不保存密钥

## 5. 典型业务链路

## 5.1 简历导入链路

1. 用户上传文件或粘贴文本
2. 前端调用 `/api/profile/import`
3. Route 解析原始文本
4. `profile-agent` 输出结构化档案
5. 前端预览并选择导入内容
6. 前端调用 `/api/profile/import/confirm`
7. Route 将数据写入 `profiles` 与 `items`

## 5.2 JD 解析链路

1. 用户输入 JD 文本
2. 前端调用 `/api/jd`
3. `jd-parser-agent` 输出结构化 JD
4. Route 存入 `job_descriptions`

## 5.3 匹配分析链路

1. 用户选择一个已解析 JD
2. 前端调用 `/api/match`
3. Route 读取默认档案的可见 Item
4. `match-agent` 计算匹配评分、差距、策略
5. Route 存入 `match_results`

## 5.4 简历生成链路

1. 用户选择叙事策略与语言
2. 可选关联 JD 与 Match Result
3. 前端调用 `/api/generate`
4. Route 读取档案、JD、匹配结果
5. `resume-gen-agent` 输出 Markdown 简历
6. Route 存入 `generated_resumes`

## 6. 目录建议理解方式

如果要快速进入代码：

- 从 `src/app/layout.tsx` 看全局布局
- 从 `src/components/Providers.tsx` 看 Provider 结构
- 从 `src/lib/hooks/*.ts` 看前端如何发请求
- 从 `src/app/api/*` 看业务入口
- 从 `src/lib/ai/*` 看 AI 调用链
- 从 `src/lib/db/*` 看持久化模型

## 7. 当前版本的技术特点

- UI 采用较强风格化的 cosmos 视觉主题
- 引入了全局 Agent 任务栏，跨页面展示 AI 任务状态
- 匹配分析结果被强制要求输出中文分析文本
- AI JSON 解析做了基础重试与截断防护
