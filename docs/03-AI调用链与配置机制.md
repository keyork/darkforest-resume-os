# 03-AI 调用链与配置机制

## 1. AI 子系统概览

AI 相关代码主要位于：

- `src/lib/ai/client.ts`
- `src/lib/ai/config.ts`
- `src/lib/ai/prompts.ts`
- `src/lib/ai/agents/*`

系统内共有四个核心 Agent：

- `profile-agent`：简历解析
- `jd-parser-agent`：JD 解析
- `match-agent`：匹配分析
- `resume-gen-agent`：简历生成

## 2. 当前配置机制

## 2.1 设计目标

当前版本的一个重要改动是：

- AI 配置不再依赖服务端 `.env`
- 改为由用户在前端页面输入
- 然后随每次请求发送给服务端

这样做的直接结果是：

- 项目部署后不必预写固定模型平台
- 同一个部署可由不同使用者连接不同 OpenAI 兼容服务
- 服务端不持久保存用户密钥

## 2.2 前端配置流

对应文件：

- `src/app/settings/page.tsx`
- `src/lib/client/ai-settings.ts`
- `src/lib/client/fetch-json.ts`

流程：

1. 用户进入 `/settings`
2. 输入 `API Key` 与 `Base URL`
3. 配置写入浏览器 `localStorage`
4. 前端请求通过请求头附带：
   - `x-ai-api-key`
   - `x-ai-base-url`

## 2.3 服务端读取配置

对应文件：

- `src/lib/ai/config.ts`

核心逻辑：

- 从请求头读取 `x-ai-api-key`
- 从请求头读取 `x-ai-base-url`
- 若缺失，抛出“请先在设置页填写”的错误

这也是为什么所有 AI 路由都需要显式传入 `request.headers`。

## 3. 通用 AI 客户端

对应文件：

- `src/lib/ai/client.ts`

## 3.1 客户端缓存

为了避免在同一进程里重复创建 OpenAI Client，系统会使用：

- `Map<string, OpenAI>`

缓存 key 为：

- `baseURL::apiKey`

这样不同用户或不同服务地址可以共存。

## 3.2 `callAgent`

用于“要求返回 JSON”的场景。

特点：

- 接受 `systemPrompt`
- 接受 `userMessage`
- 接受 `clientConfig`
- 默认模型为 `kimi-k2.5`
- 支持 `maxTokens`

## 3.3 JSON 解析保护

`callAgent` 做了几层防护：

- 兼容字符串型 content
- 兼容数组型富文本 content
- 自动去掉 ```json 代码块
- `JSON.parse` 失败时重试
- 若输出明显括号不平衡，则视为可能截断

这部分是为了解决模型有时输出半截 JSON 的问题。

## 3.4 `callAgentText`

用于返回纯文本的场景，主要服务于简历 Markdown 生成。

## 4. Prompt 设计

对应文件：

- `src/lib/ai/prompts.ts`

当前 prompt 设计原则：

- system prompt 负责定义输出结构
- route / agent 负责传入上下文
- 对 JSON 输出使用“只返回原始 JSON”强约束

### 4.1 Profile Prompt

目标：

- 从原始简历文本提取结构化个人信息、技能、经历、项目、教育、证书

### 4.2 JD Parser Prompt

目标：

- 从 JD 文本中提取职位、公司、要求、关键词、理想候选人画像

### 4.3 Match Prompt

目标：

- 输出评分、总结、需求匹配、差距分析、简历策略

额外约束：

- 分析文本必须使用简体中文

这是为了避免结果出现中英混杂。

### 4.4 Resume Generation Prompt

目标：

- 输出 Markdown 简历
- 根据策略控制叙事方式
- 可结合 JD 与匹配结果调整内容重心

## 5. 四类 Agent 的职责

## 5.1 `parseProfileFromText`

文件：

- `src/lib/ai/agents/profile-agent.ts`

职责：

- 截断过长文本
- 调用 profile prompt
- 补全空数组与默认结构

## 5.2 `parseJDFromText`

文件：

- `src/lib/ai/agents/jd-parser-agent.ts`

职责：

- 截断过长 JD
- 调用 JD parser prompt
- 规范返回对象中的可选字段和数组字段

## 5.3 `runMatchAnalysis`

文件：

- `src/lib/ai/agents/match-agent.ts`

职责：

- 把候选人资料与结构化 JD 拼接为输入
- 调用匹配分析 prompt
- 若输出分析字段没有中文，则再请求一次
- 兜底填充返回结构

## 5.4 `generateResumeMarkdown`

文件：

- `src/lib/ai/agents/resume-gen-agent.ts`

职责：

- 把可见 Item 转成模型更容易消费的文本上下文
- 拼接策略参数、JD、Match Result
- 调用文本生成接口
- 返回 Markdown

## 6. AI 请求与业务路由的关系

AI 并不是在前端直接调用第三方模型，而是通过项目自己的 API route 间接调用。

原因：

- 需要统一数据库读写
- 需要在服务端拼接上下文
- 需要控制 prompt 与输出结构
- 需要把结果持久化

因此典型执行顺序是：

前端 -> 本项目 API Route -> AI Agent -> 外部模型服务

## 7. 缺少配置时的行为

如果用户未在设置页填写配置：

- 前端请求仍会发起
- 服务端在 route 入口处读取不到配置
- 返回 `400`
- 错误信息明确提示用户到设置页配置

这种方式优于直接依赖 `.env` 的隐式失败。

## 8. 当前实现的限制

- `callAgent` 目前没有真正的 schema-level 校验，只做 JSON 解析
- 重试策略较轻，主要针对截断 JSON
- 模型名当前写死默认值，尚未做页面可配置
- 没有服务端密钥代理层或加密存储
- 所有配置都在当前浏览器本地保存，换设备后需要重新填写

## 9. 后续可演进方向

- 引入 zod schema 对 AI 输出做运行时校验
- 把模型名也放进设置页
- 加入“测试连接”按钮
- 加入 AI 请求日志与耗时统计
- 对不同 Agent 使用不同模型和不同 token 上限
