# 06 · 功能设计：投递追踪看板

## 背景与价值

用户在 darkforest 里已经完成了"制造弹药"的全流程：解析档案 → 分析 JD → 生成简历。但"把弹药打出去之后发生了什么"完全在系统之外——大多数用户用 Excel 或脑子记。

投递追踪看板的目标是**打通最后一环**：让一次完整的求职行为（发出简历 → 等待反馈 → 进入面试 → 得到结果）在同一个系统里形成闭环。

现有数据天然支持这个功能：
- `JobDescription`（JD）：投的是什么岗位
- `GeneratedResume`（生成简历）：用的是哪份简历
- `MatchResult`（匹配结果）：当时评估的匹配度是多少

## 数据模型

### `Application` 表

```ts
interface Application {
  id: string;                        // 唯一 ID
  profileId: string;                 // 关联档案（当前固定 profile_default）
  jdId?: string;                     // 关联的 JD（可选，因为用户可能直接记录没有在系统里解析过的岗位）
  resumeId?: string;                 // 关联的生成简历（可选）
  matchResultId?: string;            // 关联的匹配结果（可选）

  // 岗位信息（冗余存储，JD 被删后仍可展示）
  company: string;                   // 公司名
  position: string;                  // 岗位名
  jobUrl?: string;                   // 职位链接
  salaryRange?: string;              // 薪资范围

  // 投递状态
  status: ApplicationStatus;
  appliedAt?: string;                // 投递日期 ISO 8601
  notes?: string;                    // 备注

  createdAt: string;
  updatedAt: string;
}

type ApplicationStatus =
  | 'wishlist'      // 想投（收藏）
  | 'applied'       // 已投递
  | 'assessment'    // 笔试/测评
  | 'interview'     // 面试中
  | 'offer'         // 收到 Offer
  | 'rejected'      // 拒信
  | 'withdrawn';    // 主动放弃
```

### 状态流转

```
wishlist → applied → assessment → interview → offer
                                            ↘ rejected
                   → rejected
          → withdrawn（任意阶段可主动放弃）
```

## 页面结构

### 路由：`/applications`

#### 整体布局
```
┌─────────────────────────────────────────────────┐
│  页头：投递追踪  [+ 新增投递]                      │
├────────┬────────────────────────────────────────┤
│ 左侧   │  右侧                                   │
│ 状态筛选│  看板（按状态分列）或列表视图               │
│ + 搜索 │                                         │
└────────┴────────────────────────────────────────┘
```

#### 看板视图（Kanban）
每个状态一列，卡片可拖拽流转状态：

```
[收藏]     [已投递]    [笔试]     [面试中]   [Offer]
─────────  ─────────  ─────────  ─────────  ─────────
字节跳动    腾讯        阿里巴巴    美团       网易
后端工程师  前端工程师  算法工程师  产品经理   数据工程师
匹配 82分  匹配 71分  匹配 68分  匹配 79分  匹配 91分
昨天       3天前       1周前      2天前      今天
```

#### 卡片内容
- 公司名 + 岗位名（大字）
- 匹配度（有关联匹配结果时显示，带颜色）
- 投递日期 / 距今天数
- 关联简历版本标签
- 备注（最多 1 行）
- 右键或 hover 菜单：编辑、更新状态、删除

#### 列表视图
表格形式，支持按日期/匹配度/公司名排序。适合投递数量多时查看。

## 核心交互

### 新增投递
- 弹出 Sheet（侧边抽屉），优先从已有 JD 中选择关联，也可手动输入公司+岗位
- 可关联已生成的简历版本（下拉选择）
- 状态默认为 `applied`，可手动改
- 支持填写投递日期、职位链接、薪资范围、备注

### 状态更新
- 看板视图：拖拽卡片到目标列
- 任意视图：点卡片内的状态标签快速切换
- 进入 `interview` 状态时，可选填"面试轮次"备注

### 与现有功能联动
- 在 JD 匹配页，匹配完成后显示"加入投递追踪"快捷按钮
- 在生成简历页，生成完成后显示"记录投递"快捷按钮
- Dashboard 首页"最近匹配记录"旁边展示"待跟进投递"汇总

## API 设计

```
GET    /api/applications              # 获取所有投递记录（支持 status 过滤）
POST   /api/applications              # 新建投递记录
GET    /api/applications/[id]         # 获取单条
PUT    /api/applications/[id]         # 更新（状态、备注等）
DELETE /api/applications/[id]         # 删除
PATCH  /api/applications/[id]/status  # 快速更新状态
```

## 实现顺序建议

1. Drizzle schema + 迁移脚本
2. `/api/applications` CRUD 路由
3. React Query hooks（`useApplications`, `useCreateApplication` 等）
4. 列表视图（比 Kanban 简单，先上）
5. 新增/编辑 Sheet 表单
6. Kanban 视图 + 拖拽（用 `@dnd-kit/core`，项目中已有依赖）
7. Dashboard 联动展示
8. 匹配页、生成页快捷入口
